Wordle DSL — Complete Architecture Guide

Hinglish • JSON examples • Code walkthrough • Migration plan • Real Wordle analogies

1. DSL Kya Hai?

DSL = Domain-Specific Language. Ek chhota JSON-based "mini language" hai jo Wordle ke game rules describe karta hai bina TypeScript code likhe.

Asli baat: AI ek user prompt padh ke JSON rules generate karti hai, server validate karta hai, RTDB save karta hai, client interpret karke runtime pe execute karta hai.

Soch lo recipe cards:
  • Pizza recipe = ek DSL rule list ("agar oven pe 200°C ho aur 12 min beet jaaye, to pizza nikal lo")
  • Recipe card = JSON file (paper pe steps)
  • Chef = DSL Engine (recipe padh ke kitchen mein execute karta hai)
  • Naya pizza chahiye? Bas nayi recipe card likho — kitchen wapas se nahi banana padta

Aaj ki taarikh tak Wordle mein 4 chizein DSL-driven hain: events (kab kuch trigger ho), conditions (kya check karna hai), operators (kaise compare karna hai), aur actions (kya karna hai).

2. DSL vs Composable Primitives — Kya Farak Hai?

App mein do alag approaches use hote hain. Ship Fight Composable Primitives use karta hai. Wordle True DSL use karta hai.

🔹 Composable Primitives🔸 True DSL
Storage "logic" blob — primitive name reference "dsl" blob — pura rule JSON
AI ka kaam Pre-coded functions ke whitelist mein se choose karna Free-form JSON rule generate karna
Naya behavior kaise aaye? ❌ Developer ko TypeScript code likhna padega + deploy ✅ Bas naya prompt — koi code change nahi
AI flexibility Limited — sirf pre-defined options High — params (timer seconds, hint count, etc.) freely choose
Type safety Strong (TS compile-time) Runtime validation (Gemini schema + server validator)
Complex per-game logic Better (custom code per primitive) Limited to engine's action vocabulary
Use case fit Complex turn-based games (Ship Fight) Rule-driven games (Wordle, Hangman)
Primitives = Restaurant menu: AI sirf existing dishes order kar sakta hai. Naya dish chahiye? Chef ko naya recipe seekhana padega (code change).

DSL = Home cooking with shared pantry: AI bata sakta hai "le 2 onion + 1 tsp salt + 5 min boil". Pantry items (events/actions/fields) fixed hain, lekin combination infinite.

Kya Primitives mein POSSIBLE NAHI tha jo DSL mein hai?

  • Conditional triggers: "agar wrongGuessCount >= 3 to..." — Primitives mein har condition ke liye alag function chahiye
  • AI param tuning: AI freely 30s, 45s, 60s choose kar sakti hai. Primitives mein setTimer30s() aur setTimer60s() alag functions chahiye
  • Action chaining: Ek event pe 3 actions (timer + hint + extra guess) — Primitives mein combination ke liye nayi primitive likhni padti
  • Runtime distribution: Variant JSON RTDB se aata hai. Bina app update doosre users ke phone pe naya rule kaam karta hai
  • AI invention: Wo combinations jo developer ne soch bhi nahi liye — AI khud explore kar sakti hai

3. DSL ke 4 Building Blocks

Pura DSL sirf 4 cheezon ka game hai. In char ko mila ke infinite variations bante hain.

// Pseudo-code from lib/dsl/game_dsl.dart
List<DSLAction> evaluate(String event, DSLEvalContext ctx) {
  final matched = [];
  for (final trigger in dsl.activeTriggers) {
    if (trigger.event != event) continue;
    if (trigger.conditions.every((c) => c.op.evaluate(ctx.valueOf(c.field), c.value))) {
      matched.addAll(trigger.actions);
    }
  }
  return matched;
}

9d. Action dispatch

void _applyDslActions(List<DSLAction> actions) {
  var s = gameState.value;
  for (final a in actions) {
    switch (a.type) {
      case WordleActions.addGuesses:
        s = s.copyWith(extraGuesses: s.extraGuesses + count);
      case WordleActions.setGuessTimer:
        s = s.copyWith(guessTimerSec: secs);
      case WordleActions.revealHint:
        s = s.copyWith(hintsGiven: s.hintsGiven + 1, pendingHint: true);
      case WordleActions.skipGuess:
        s = s.copyWith(currentInput: '');
      case WordleActions.endGame:
        s = s.copyWith(phase: outcome == 'win'
            ? WordlePhase.won : WordlePhase.lost);
    }
  }
  gameState.value = s;
}

12. Impossible Game Banane Pe Kya Hota Hai?

Agar user kuch aisa prompt deta hai jo DSL support nahi karta — to system gracefully handle karta hai. Validation 3 layers pe hoti hai.

Layer 1: Gemini Schema (AI ka boundary)

Gemini ko responseSchema ke through forced structured output milta hai. Schema mein sirf valid event/action/field names hain. Gemini physically galat enum value nahi de sakti.

Example: User bole — "After 2 wrong guesses, add tomato sauce". Gemini reply mein reply: "Sorry, I can't add sauce to Wordle :)" dega aur DSL mein wahi pre-defined actions mein se nearest match (probably no trigger or revealHint) generate karega.

Layer 2: Server Validation (validateWordleDSL)

Even agar Gemini somehow weird JSON bhej de — server filter karta hai:

// Pseudo from wordle_dsl_engine.ts
function validateTrigger(raw) {
  if (!DSL_EVENTS.includes(raw.on)) return null;     // drop unknown event
  conditions = (raw.conditions || [])
    .map(validateCondition)
    .filter(c => c !== null);                       // drop bad fields/ops
  actions = (raw.actions || [])
    .map(validateAction)
    .filter(a => a !== null);                       // drop bad action types
  if (actions.length === 0) return null;             // no-op trigger discarded
  return { on, conditions, actions };
}

Result: invalid triggers silently drop ho jaate hain. User ko ek partial DSL milti hai jo valid hi hai.

Layer 3: Client Engine

Client side WordleController._applyDslActions() mein default case hai jo unknown action types pe break karta hai — koi crash nahi.

Examples of impossible prompts & system response

Architecturally important: System NEVER crashes on bad input. Worst case = empty DSL = base Wordle gameplay. Best case = AI partial implementation. User ko hamesha kuchh playable milta hai.
