🍳 DSL Architecture — The Cloud Kitchen Story

How it will work when all games are migrated — English food analogy + file structure

1. Cloud Kitchen Setup — Imagine This

Imagine a large cloud kitchen building where 5 different brands cook food from the same place:
🍕 Pizza Hut
= Wordle
🍛 Biryani House
= Ship Fight
🍣 Sushi Express
= Red Flag
🍢 Tandoori Tales
= Emoji Guess
🍔 Burger King
= Tic Tac Toe
The entire setup is in a shared kitchen, but each brand has its own separate menu.

This is the exact structure of your DSL approach — 5 games run on a shared engine, but each has its own unique flavor (events, actions, fields).

Key insight: Build the common infrastructure once, and then every brand takes advantage of it without compromising on their specific menus.

2. The Building Structure

The building has 3 floors. Let's go from bottom to top:

G
Ground Floor — Shared Kitchen
Universal infrastructure • Used by all brands

Stove, gas, knives, timer, plates — built once, shared by everyone.

1
First Floor — Per-Brand Menus
Each brand's own menu card

Pizza Hut has its own menu, Biryani House has its own menu — brand-specific recipes + ingredients.

2
Second Floor — FSSAI Inspector (Server)
Quality control • Every dish is checked

Before reaching the customer's house, every order is inspected. Cheating-proof.

3. Floor 1 — Shared Kitchen (Generic DSL Core)

This is the common infrastructure for everyone. All brands use this.

lib/dsl/ ← Ground floor (shared) ├── game_dsl.dart ← Gas stove + pan (engine) ├── dsl_spec.dart ← Recipe-card template └── widgets/ └── dsl_status_chips.dart ← Timer, plates, scale (UI)
File Food equivalent Used by all brands?
game_dsl.dart 🔥 Gas stove + pan + pot ✅ Universal
dsl_spec.dart 📋 Menu format / recipe-card template ✅ Universal
dsl_status_chips.dart ⏱️ Timer, 🍽️ plates, ⚖️ scale ✅ Universal

This code was built during Wordle — now every game will use it for free

The Pizza Hut guy will use the timer, the Biryani guy too, and the Sushi guy as well. Build once, benefit 5 times.

// lib/dsl/game_dsl.dart — universal cooking logic
class DSLEngine {
  final GameDSL dsl;
  const DSLEngine(this.dsl);

  List<DSLAction> evaluate(String event, DSLEvalContext ctx) {
    // Universal evaluation — the engine doesn't know if this is a pizza or biryani
    final matched = <DSLAction>[];
    for (final trigger in dsl.activeTriggers) {
      if (trigger.event != event) continue;
      if (trigger.conditions.every((c) => c.passes(ctx))) {
        matched.addAll(trigger.actions);
      }
    }
    return matched;
  }
}

4. Floor 2 — Per-Brand Menus (Per-Game Specs)

Every game has its own separate menu file. This is the "brand identity".

lib/models/ ← First floor (per-brand) ├── wordle_dsl.dart 🍕 Pizza Hut menu ├── ship_fight_dsl.dart 🍛 Biryani House menu (future) ├── red_flag_dsl.dart 🍣 Sushi Express menu (future) ├── emoji_guess_dsl.dart 🍢 Tandoori Tales menu (future) └── tictactoe_dsl.dart 🍔 Burger King menu (future)

What is inside each menu file?

Brand Own Events (when to fire) Own Actions (what to do)
🍕 Pizza Hut (Wordle) wrongGuess, correctGuess, timerExpiry addGuesses, setGuessTimer, revealHint
🍛 Biryani House (Ship Fight) turnStart, shipDestroyed, playerHit damageShip, healShip, spawnPickup
🍣 Sushi Express (Red Flag) questionShown, cardFlipped flipCard, showHint, endRound
🍔 Burger King (Tic Tac Toe) marked, roundEnd, tieDetected swapTurn, disableCell, resetBoard
Pizza Hut's menu doesn't have a "biryani masala" item. Biryani House's menu doesn't have "extra mozzarella".
Logical separation — each brand is safe in its own world.

5. Floor 3 — Server-side FSSAI Inspector

The exact same structure exists on the server too — a shared core + per-game files.

functions/src/ ├── dsl/ ← Shared inspector (future refactor) │ └── engine.ts ← Universal validation logic │ └── controllers/ ├── wordle_dsl_engine.ts 🍕 Pizza Hut FSSAI rules ├── ship_fight_dsl_engine.ts 🍛 Biryani House FSSAI rules (future) └── ...
The FSSAI Inspector looks at every order and checks:
  • Is this item on the brand's menu? (event/action valid?)
  • Is the quantity correct? (param within range?)
  • Is the recipe complete? (trigger has actions?)
The inspection process is universal — just the menu reference is specific to each brand.

Even if Gemini AI generates some garbage, the inspector filters it out. The customer always gets valid food.

6. How Everything Works Together

Imagine: The customer ordered — "Pizza Margherita with extra cheese after 5 min wait":

1
Customer (User) — Wrote prompt on the app: "extra cheese after 5 min wait"
2
Order App (AI Chat) — Identified cuisine → picked the Pizza Hut menu
3
Recipe Card is created (DSL JSON) — Only valid items from the Pizza Hut menu (addCheese is valid)
4
Inspector checks it (validateWordleDSL) — Cross-checks with the Pizza Hut menu
5
Cooking in the Shared Kitchen (DSL Engine evaluate) — Stove (engine) follows the recipe
6
Wait using the Universal Timer (DslCountdownChip) — The same timer the Biryani guy uses
7
Pizza ready → serve (Action apply) — The player gets a cheese-loaded pizza
Steps 1, 2, 4 use Pizza Hut's specific menu.
Steps 3, 5, 6, 7 use the shared kitchen tools.
A single change in the menu only affects the pizza — not the biryani or sushi.

The pattern is clear: Every file has a single-purpose role. Shared things are in lib/dsl/, brand-specific things are in lib/models/<game>_dsl.dart.

9. Practical Implications

9a. How easy is it to add a new game?

Pizza Hut is already set up (Wordle done). Want to open Biryani House?
  • Just write ship_fight_dsl.dart (menu)
  • Just write ship_fight_dsl_engine.ts (server FSSAI)
  • 5-6 lines of integration in the existing controller + screen
No need to touch the shared kitchen (engine, widgets) at all. The stove is already installed. Just write the new menu.

9b. Will a bug fix in one game affect another?

No. Fixing a typo in Pizza Hut's menu doesn't change the Biryani menu.

But — if you fix a bug in the shared kitchen (engine), all brands will benefit. Fix in one place, effect in 5 places. This is the real advantage of the DSL.

9c. Can UI components be shared?

Yes, anything inside lib/dsl/widgets/ (timer chip, toast helper, badge) — any game can use it directly. Universal cutlery.

9d. Can there be a conflict between 2 games?

No, because every game uses its own string constants:

  • Wordle: WordleEvents.wrongGuess
  • Ship Fight: ShipFightEvents.shipMove

The engine doesn't care which game the event belongs to — it just checks the event's string name and finds the matching trigger.

10. One-Line Summary

DSL approach = a shared cloud kitchen where 5 different brands run their own menus.

Common infrastructure (stove, timer, plates) is shared by everyone — each brand has its own unique flavor (events, actions).

Adding a new brand is easy, a change in one brand doesn't impact the other brands, and improvements in shared tools benefit all brands.
3 main takeaways:
  1. 🏗️ Build once, share forever — Shared kitchen is built once, useful forever
  2. 🎯 Brand isolation — Pizza Hut's problems don't reach Biryani House
  3. 🚀 Migration cost drops dramatically — First game took 22 files, subsequent games take 6 files
