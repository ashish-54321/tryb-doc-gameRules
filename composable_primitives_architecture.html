Composable Primitives Architecture

Phase 3 — making Ship Fight infinitely customizable without parameter explosion

Tryb · System Design
Approach comparison & rationale
4 starter modifiers shipped
★ The problem we solved

Phase 1 & 2 added fixed parameters (cosmetic fields, rule numbers, arena layout). Every new user idea required a developer to add a new parameter. Once user creativity outpaced the schema, requests started getting denied: "engine doesn't support that." Phase 3 replaces this with a plug-in primitive system — AI composes behaviors from a registered library, and adding new ones is a single-file change.

1 What was shipped — the logic blob

A third typed blob alongside cosmetic + rules at /gameLibrary/{id}. Holds a list of modifier primitives that the engine applies per tap.

// New field on every Ship Fight variant entry "logic": { "modifiers": [ { "type": "fatigue", "params": { "decayRate": 0.85, "minMultiplier": 0.4, "recoveryMs": 1500 } }, { "type": "momentum", "params": { "boostPerStreak": 0.05, "maxBoost": 1.5, "windowMs": 250 } }, { "type": "streakBonus", "params": { "every": 10, "multiplier": 2.0 } } ] }

2 The 4 starter modifier primitives

🥵fatigue
Rapid spam drains a per-player energy meter; rest recovers it. Push effectiveness scales with energy. Endurance feel.
⚡momentum
Consecutive fast taps (within a window) build a multiplier up to a cap. Rewards rhythm.
🔥streakBonus
Every Nth consecutive tap deals a big bonus push. Combo feel.
💥panicBurst
When pushed close to losing, opens a brief boosted-push window. Drama / comeback.

The 5 Approaches We Compared

From rigid fixed schema to free-form code generation — what we considered

Trade-off analysis
Safety vs flexibility vs cost
1
Fixed Parameter Schema PREVIOUS
Every customizable thing is a named field with a fixed type. Developer adds a parameter; AI fills it. Used in Phase 1 & 2 (cosmetic fields, rules, arenaLayout).

Pros

  • Strongly typed end-to-end — IDE autocomplete, refactor-safe
  • Trivially server-validated — clamp/enum each field
  • Engine rendering predictable — no surprise inputs
  • Zero risk to anti-cheat
  • Easy to debug — flat shape

Cons

  • Hard ceiling — anything not in schema is impossible
  • Each new idea = developer task + redeploy
  • Schema grows monotonically — no easy way to remove unused fields
  • User intent often doesn't fit a single field
  • Combinatorial explosion as variants grow
2
Composable Primitives CURRENT · RECOMMENDED
A registry of typed primitives ("Lego blocks") — modifiers, layouts, win conditions. AI picks & configures combinations. Used from Phase 3 onward.

Pros

  • Adding a new primitive = 1-file change
  • AI composes infinite combinations from finite blocks
  • Each primitive independently typed + validated
  • Server stays fully authoritative (whitelist + clamps)
  • Engine logic stays explicit — no interpreter to maintain
  • Variants degrade gracefully — unknown primitives dropped

Cons

  • Still bounded by the registry — exotic ideas still need a dev
  • Modifier interactions need thought (multiplicative stacking)
  • Per-primitive scratchpad adds RTDB writes per tap
3
Domain-Specific Language (DSL) FUTURE OPTION
AI emits a small declarative "rules program" — events, conditions, effects. Engine has an interpreter that runs it inside a sandbox.

Pros

  • Maximum flexibility — AI can express genuinely new mechanics
  • Compose conditions/effects orthogonally
  • Engine becomes a generic runtime for many game styles

Cons

  • Interpreter complexity — 3–4 weeks of engineering
  • Edge cases / infinite loops need timeout enforcement
  • Debugging emergent bugs is painful
  • Mobile CPU overhead for interpretation
  • AI failure modes harder to predict
4
AI-generated Sandbox Code NOT RECOMMENDED
AI writes actual JavaScript/Lua that the engine sandboxes and executes per tap.

Pros

  • Theoretically unlimited expressiveness
  • One mechanism handles every game ever conceived

Cons

  • Anti-cheat collapses — server can no longer reason about game state
  • Sandbox escapes are an ongoing security war
  • Mobile performance hit (V8 / interpreter startup)
  • Code review impossible at scale — content moderation nightmare
  • Cost: every match runs LLM-generated code → unpredictable bills
5
Curated Template Library ALTERNATIVE
Engineers ship a fixed set of "game templates" (Tug of War, Race, Brawl…). AI's only job is to pick a template + tune its few exposed knobs.

Pros

  • Each template hand-tuned for great feel
  • Simplest possible safety story
  • Onboarding/UX easier (browseable catalog)

Cons

  • Limited variety — capped at N templates
  • Doesn't scale with community creativity
  • Two templates = effectively two engines to maintain
  • Hybrid ideas (race + comeback boost?) fall through cracks

Why Composable Primitives Wins

5 Decision matrix

Each approach scored on the 5 axes that matter for this app. Higher = better.

✓ Verdict

Composable Primitives is the only approach that scores high on all 5 axes simultaneously. It keeps the safety story of fixed schema (whitelist + clamp), the extensibility of a DSL (1-file primitive additions), and the engine cost of a template library (explicit branches, no interpreter overhead). DSL is a tempting future jump but premature today — our biggest constraint is user-demand signal, not engine flexibility.

6 Why this architecture wins

♾️

Finite blocks, infinite combos

4 modifiers × 0–3 per variant = thousands of feel combinations from one engine.

🛡️

Server stays the source of truth

Every primitive is registered, validated, clamped. AI literally cannot inject behavior we didn't pre-approve.

📂

Adding a primitive = 1 file

Schema + validator + apply function — one entry in MODIFIER_PRIMITIVES. AI system prompt updates automatically.

🔁

Backwards-compatible by default

Variants without a logic blob behave exactly like classic Ship Fight. Unknown primitive types in old data are silently dropped.

📊

Analytics-friendly

Which primitives drive retention? Trivial to A/B test — they're typed and counted in RTDB.

🚀

DSL is still on the table

If user creativity ever outruns this registry, we can ship a DSL on top of the primitives — they remain the safe runtime.

★ Bottom line

We moved from "what does the developer let you do" to "what can the developer's blocks compose into." The ceiling didn't disappear — it just got pushed far higher, and now it raises by adding files instead of editing every existing file. When a future user idea hits the new ceiling, we add one more primitive instead of redesigning the engine.
