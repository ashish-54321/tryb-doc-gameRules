Wordle DSL — Complete Architecture Guide

English • JSON examples • Code walkthrough • Migration plan • Real Wordle analogies

1. What is a DSL?

DSL = Domain-Specific Language. It is a small JSON-based "mini language" that describes Wordle game rules without writing TypeScript code.

The core idea: The AI reads a user prompt and generates JSON rules, the server validates it, the RTDB saves it, and the client interprets and executes it at runtime.

Think of recipe cards:
  • Pizza recipe = a DSL rule list ("if the oven is at 200°C and 12 minutes pass, take out the pizza")
  • Recipe card = JSON file (steps on paper)
  • Chef = DSL Engine (reads the recipe and executes it in the kitchen)
  • Want a new pizza? Just write a new recipe card — you don't need to rebuild the kitchen from scratch.

As of today, 4 things in Wordle are DSL-driven: events (when to trigger), conditions (what to check), operators (how to compare), and actions (what to do).

2. DSL vs Composable Primitives — What's the Difference?

The app uses two different approaches. Ship Fight uses Composable Primitives, while Wordle uses a True DSL.

🔹 Composable Primitives 🔸 True DSL
Storage "logic" blob — primitive name reference "dsl" blob — full rule JSON
AI Function Selects from a whitelist of pre-coded functions Generates free-form JSON rules
How is new behavior added? ❌ Developer must write TypeScript code + deploy ✅ Just a new prompt — zero code changes
AI Flexibility Limited — predefined options only High — params (timer seconds, hint count, etc.) can be freely chosen
Type Safety Strong (TS compile-time) Runtime validation (Gemini schema + server validator)
Complex Per-Game Logic Better (custom code per primitive) Limited to engine's action vocabulary
Use Case Fit Complex turn-based games (Ship Fight) Rule-driven games (Wordle, Hangman)
Primitives = Restaurant menu: The AI can only order existing dishes. Want a new dish? The chef needs to be taught a new recipe (code change).

DSL = Home cooking with a shared pantry: The AI can specify "take 2 onions + 1 tsp salt + boil for 5 min". The pantry items (events/actions/fields) are fixed, but the combinations are infinite.

What was NOT POSSIBLE with Primitives that is possible with DSL?

  • Conditional triggers: "if wrongGuessCount >= 3 then..." — In Primitives, every unique condition requires a separate function.
  • AI parameter tuning: The AI can freely choose 30s, 45s, or 60s. In Primitives, you would need separate functions like setTimer30s() and setTimer60s().
  • Action chaining: 3 actions on a single event (timer + hint + extra guess) — In Primitives, you would have to write a completely new primitive for this combination.
  • Runtime distribution: The variant JSON comes from the RTDB. The new rule works on other users' phones without needing an app update.
  • AI invention: The AI can explore combinations that the developer might not even have thought of.

3. The 4 Building Blocks of DSL

The entire DSL is built on just 4 building blocks. Combining these four creates infinite variations.

3a. Events 5 total — When do they fire?

Event When it fires Example rule
gameStart As soon as the game begins Set timer / grant extra guesses
wrongGuess After submitting an incorrect guess Reveal hint after 2 wrong guesses
correctGuess Upon guessing the correct answer Bonus action / celebration
guessSubmit After every guess submission (both correct/incorrect) Per-guess timer reset
timerExpiry When the per-guess timer reaches 0 Skip guess / lose 1 attempt

3b. Condition Fields 4 total — What to check?

Field What it tracks
wrongGuessCount How many wrong guesses have been made so far
guessNumber Which guess number this currently is (1, 2, 3...)
hintsGiven How many hints have been revealed
remainingGuesses How many guesses are remaining

3c. Operators 6 total — How to compare?

Operator Meaning Example
eq Equal (==) wrongGuessCount == 2
neq Not equal (!=) guessNumber != 1
gt Greater than (>) wrongGuessCount > 3
gte Greater or equal (>=) wrongGuessCount >= 2
lt Less than (<) remainingGuesses < 2
lte Less or equal (<=) remainingGuesses <= 1

3d. Actions 5 total — What to do?

Action What it does Params Server enforced?
addGuesses Grants extra guesses count: 1-3 ✅ Yes
setGuessTimer Sets a countdown timer per guess seconds: 10-120 ⚠️ Client only
revealHint Displays a hint toast (none) ⚠️ Client only
skipGuess Clears the current input (none) ⚠️ Client only
endGame Forces the game to end outcome: "win"/"lose" ⚠️ Client only

4. JSON Input/Output Shape

The entire DSL is a single JSON object. Here is the complete template:

{
  "triggers": [
    {
      "on":         "<event-name>",
      "conditions": [
        {
          "field": "<field-name>",
          "op":    "<operator>",
          "value": <number>
        }
      ],
      "actions": [
        {
          "type":   "<action-type>",
          "params": { "...": "..." }
        }
      ]
    }
  ]
}

Simple Real Example

The user said: "After 2 wrong guesses, give me 1 extra guess and reveal a hint."

{
  "triggers": [
    {
      "on": "wrongGuess",
      "conditions": [
        { "field": "wrongGuessCount", "op": "gte", "value": 2 }
      ],
      "actions": [
        { "type": "addGuesses",  "params": { "count": 1 } },
        { "type": "revealHint" }
      ]
    }
  ]
}

11

Runtime evaluation

The engine evaluates the triggers on every event:

// Pseudo-code from lib/dsl/game_dsl.dart
List<DSLAction> evaluate(String event, DSLEvalContext ctx) {
  final matched = [];
  for (final trigger in dsl.activeTriggers) {
    if (trigger.event != event) continue;
    if (trigger.conditions.every((c) => c.op.evaluate(ctx.valueOf(c.field), c.value))) {
      matched.addAll(trigger.actions);
    }
  }
  return matched;
}
12

Actions are applied

Matched actions are dispatched. The UI state updates → the timer chip shows up, a hint toast appears, and the counter updates to 2/7.

File: lib/controllers/wordle_controller.dart → _applyDslActions()

Server-side cap enforcement

submitWordleGuess also replays the DSL on the server side — if a player submits a 7th guess and the DSL has set addGuesses: 1, then computeEffectiveMaxGuesses() returns 7 and the guess is accepted. Cheating-proof.

9d. Action dispatch

void _applyDslActions(List<DSLAction> actions) {
  var s = gameState.value;
  for (final a in actions) {
    switch (a.type) {
      case WordleActions.addGuesses:
        s = s.copyWith(extraGuesses: s.extraGuesses + count);
      case WordleActions.setGuessTimer:
        s = s.copyWith(guessTimerSec: secs);
      case WordleActions.revealHint:
        s = s.copyWith(hintsGiven: s.hintsGiven + 1, pendingHint: true);
      case WordleActions.skipGuess:
        s = s.copyWith(currentInput: '');
      case WordleActions.endGame:
        s = s.copyWith(phase: outcome == 'win'
            ? WordlePhase.won : WordlePhase.lost);
    }
  }
  gameState.value = s;
}

9e. UI reflects state

WordleScreen watches the gameState using a ValueListenableBuilder. When the state changes:

  • state.guessTimerSec != null → cyan DslCountdownChip renders
  • state.pendingHint == true → showDslToast() displays "💡 Hint revealed!"
  • state.maxGuesses = 6 + extraGuesses → counter updates to "2/7"
  • state.phase changes → triggers the win/lose overlay

12. What Happens if an Impossible Game is Created?

If a user submits a prompt that the DSL doesn't support, the system handles it gracefully. Validation happens across 3 separate layers.

Layer 1: Gemini Schema (The AI Boundary)

Gemini is forced into a structured output format via the responseSchema. The schema only contains valid event/action/field names. Gemini physically cannot output an invalid enum value.

Example: If a user says — "After 2 wrong guesses, add tomato sauce". Gemini will answer with reply: "Sorry, I can't add sauce to Wordle :)" and will generate a DSL matching the closest valid predefined action (probably no trigger or revealHint).

Layer 2: Server Validation (validateWordleDSL)

Even if Gemini somehow sends weird JSON — the server filters it out:

// Pseudo from wordle_dsl_engine.ts
function validateTrigger(raw) {
  if (!DSL_EVENTS.includes(raw.on)) return null;     // drop unknown event
  conditions = (raw.conditions || [])
    .map(validateCondition)
    .filter(c => c !== null);                       // drop bad fields/ops
  actions = (raw.actions || [])
    .map(validateAction)
    .filter(a => a !== null);                       // drop bad action types
  if (actions.length === 0) return null;             // no-op trigger discarded
  return { on, conditions, actions };
}

Result: Invalid triggers are silently dropped. The user receives a partial DSL that remains fully valid.

Layer 3: Client Engine

The client-side WordleController._applyDslActions() contains a default case that simply performs a break on unknown action types — preventing any crashes.

Examples of impossible prompts & system response

User prompt System Response
"Add 10 extra guesses" Clamped: Only counts of 1-3 are allowed. The AI will likely generate count: 3.
"500 second timer" Clamped: Seconds must be 10-120. Final value: 120.
"After 20 wrong guesses end game" Condition value capped at 100 (sanity check). Trigger is valid but will never fire.
"Change tile colors to blue mid-game" No matching action. Gemini provides an explanation in its reply, and no DSL trigger is generated.
"Play music when user is offline" No matching event/action. Results in an empty DSL or just reply text.
"Order a pizza during the game" Refusal in the reply, empty DSL.
"100 triggers all at once" Max 8 triggers allowed — extras are dropped.
Architecturally important: The system NEVER crashes on bad input. Worst case = empty DSL = standard baseline Wordle gameplay. Best case = partial AI implementation. The user always gets something playable.

  • ❌ Custom counters / cross-event state

One-line takeaway

Wordle DSL is a mini rule-engine — define gameplay behavior by writing JSON, where the AI generates it, the server validates it, and the client interprets and executes it at runtime. New behavior = new prompt. No app update required, ever.
